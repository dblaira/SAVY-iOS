Bottom line: Understood is not competing with Re_Call — it feeds it. The axiom-style Postgres ontology in Understood is valuable corpus, not the final authority layer — it was built before W3C formal tools existed in your workflow. Formal meaning lives in OWL/Turtle (Protégé) + validated triples (Docker/W3C stack) + Re_Call's graphTrace engine. No app gets its own orphan ontology. No AI path without a traceable triple.

Two systems, two eras — don't confuse them

Understood.app ontology (2024–2025): personal if-then rules in Postgres, human-confirmed axioms, ~5k–8k extractions — built to govern AI prompts before you had Protégé and the W3C RDF pipeline. It works. It is not OWL in the database, and it should not be narrated as "the best system." Re_Call + W3C stack (2025–2026): OWL classes authored in Protégé, triples validated in Docker (riot + SPARQL + HermiT), deterministic n3 engine, graphTrace on every AI path — interoperability by design, hallucination curb by architecture. The suite plan: migrate corpus → formal triples → one engine → all surfaces cite paths.

Understood axioms + extractions → Formal RDF triples (shared) → Re_Call engine + graphTrace → All apps cite paths

Why this architecture exists

From your Obsidian vault: memory compresses life into stories; life runs on delayed, connected effects. The suite is built for the second problem, not the first.

🗯 Memory is narrative

Storytelling brain

Your mind turns life into stories — "bad week because work was stressful." Memory compresses, edits, and explains after the fact. Most apps stop here: prose, summaries, coaching copy.

🔥 Life is relational

Connected effects

Sleep changes ambition. Learning changes mood. Belief changes entertainment two weeks later. Exercise + sleep may do something neither does alone. Understood reveals connections; Re_Call recovers relational context at return moments.

You're using a storytelling brain to understand a life made of delayed, connected effects. The deterministic ontology layer exists so AI assists that relational life — without inventing facts, without crowd-generalized advice, without silent mutation of your meaning.

How ontologies are built vs how triples are validated

Ontology = the contract (what kinds of things and relations are allowed). RDF triples = the state (actual facts). Different tools, different jobs — both W3C-aligned.

Author meaning
Protégé (macOS)

OWL classes, object/datatype properties, imports (BFO/CCO), competency questions. HermiT reasoner checks logical consistency. Source: .ttl files humans steward.

Validate facts
Docker + W3C stack

Jena riot (syntax) · SPARQL (query proofs) · N-Triples canonicalization · ontology/validate.sh in Re_Call. Interoperability: same triples work across tools, browsers, apps.

Run at request time
n3 + graphTrace

In-memory graph walk over Turtle rules. Deterministic match → rank → graphTrace payload. LLM receives graph JSON only — never invents eligibility or facts.

Privacy & personalization shield

Shielded from generalized feedback

  • Deterministic layer decides eligibility — not "what users like you do"
  • Reactive recommendations stay within revealed strength — deeper, not broader
  • LLM drafts language only; graph decides what is allowed
  • Agent refuses unsupported personal inference — shows rule used or says "don't know yet"

Safe to add more data

  • Personal graph IRI: understood.app/graph/personal — your triples, your rules
  • Provenance on every axiom: supportedBy entry evidence, human confirmation
  • RLS + Cognito/Supabase auth — no cross-user graph leakage
  • Inspectable: templates, rules, feedback you control — not black-box coaching

Design principle (W3C): interoperability prevents digital fragmentation — documents, programs, and web specs work identically across devices. Applied here: one IRI tree, one triple shape, one graphTrace contract — so your meaning survives every app surface.

Vocabulary

RDF triples, Shackle, and Sparkle — mapped to the actual technologies:

RDF triples
Subject → predicate → object. e.g. ScanCalendarReminder → revealsStrength → TimeAwareness. Core of Re_Call.
SPARQL (likely “Sparkle”)
Query language that walks the graph. Used in ontology/validate.sh and ontology/queries/*.rq for proofs — not runtime on Vercel.
SHACL (likely “Shackle”)
Shape constraints — “this graph must look like X.” Scaffold exists in understood-app; not enforced in Re_Call DB yet.
Original Spark
Product doctrine term (narrative vs relational memory) — not a technology.
Aurora
AWS managed PostgreSQL. SAVY's beliefs table + correlation JSON snapshots. Storage — not meaning authority by itself.
Neo4j
Graph DB for traversal. Re_Call projects to it; SAVY uses it for correlation edges. Read surface — not source of truth in either app today.

Why Aurora alone doesn't curb hallucinations

What Aurora gives SAVY

  • Relational rows with UUIDs (beliefs)
  • One latest analysis row with correlations as JSON array
  • Projection views for Neo4j export

Traceable at the row level. Not at the rule + path level AI needs.

What hallucination control requires

  • Every claim links to triple IDs
  • Engine proves which rule authorized the claim
  • Model receives graphTrace — invents nothing
  • User can audit: "show me the path"

That's Re_Call's stack — not a database brand.

The Vercel ontology 504 fix (Neo4j timeout → Aurora fallback) is a reliability patch. It does not add meaning authority. Same snapshot, different read path.
